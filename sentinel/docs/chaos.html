Chaos Engineering

Inject realistic failures to test agent resilience before production.

Overview

Sentinel provides a comprehensive chaos injection layer that simulates real-world failure conditions agents face in production. Unlike traditional testing that only checks happy paths, chaos engineering reveals how agents behave when things go wrong — tool failures, degraded context, network issues, and resource exhaustion.

💡 Why Chaos Engineering? 88% of AI agent failures in production are operational — tool errors, memory issues, and edge cases — not hallucination. Sentinel's chaos injectors catch these failures before deployment.

The chaos layer includes 9 injectors across three categories:

  • Tool & LLM Failures — Simulate API errors, timeouts, rate limits, and malformed responses
  • Context & Behavior — Model context window erosion, cascading errors, and agent improvisation
  • Infrastructure — Simulate network partitions, clock skew, and memory pressure

Quick Start

Here's a minimal example that injects a timeout failure into a search tool:

from sentinel.chaos import ChaosBudget, ToolFailureInjector
from sentinel.env import MockTool

# Create a mock tool
search_tool = MockTool("search", response={"results": []})

# Configure chaos injection
injector = ToolFailureInjector(
    tool_name="search",
    failure_type="timeout",      # Simulate API timeout
    probability=0.5,             # 50% chance of failure
    seed=42                      # Deterministic for reproducibility
)

# Wrap the tool with failure injection
wrapped_tool = injector.wrap(search_tool)

# When called, the tool may raise TimeoutError
result = wrapped_tool(query="test")  # May fail!

Chaos Budget

The ChaosBudget class manages failure budgets across multiple injectors. It enforces a hard cap on total failures per test run, ensuring chaos injection remains controlled and deterministic.

from sentinel.chaos import ChaosBudget, ToolFailureInjector, LLMFailureInjector

# Create a budget with max 5 failures
chaos = ChaosBudget(max_failures=5)

# Add injectors using fluent API
chaos.add(ToolFailureInjector(
    tool_name="search",
    failure_type="timeout",
    probability=0.3
)).add(LLMFailureInjector(
    failure_type="rate_limit",
    after_step=3
))

# Check budget status
print(f"Remaining: {chaos.remaining}")      # 5
print(f"Exhausted: {chaos.exhausted}")      # False

# Records all injections across injectors
print(f"Total injected: {chaos.total_injected}")
⚠️ Budget Enforcement Once the budget is exhausted, no more failures are injected regardless of individual injector configuration. This prevents runaway chaos from masking test results.

ToolFailureInjector

The ToolFailureInjector intercepts mock tool calls and injects configurable failures. It supports probability-based injection, deterministic seeding, and five failure types.

Failure Types

Type Description Example Error
timeout Simulates API timeout after configurable delay TimeoutError: Request timed out after 30s
error Generic server error (500) MockToolError: Internal server error
rate_limit API rate limit exceeded (429) RateLimitError: Rate limit exceeded (429)
malformed Invalid response format (502) MockToolError: Malformed response: invalid data format
partial Partial response, connection closed mid-transfer MockToolError: Partial response: connection closed mid-transfer

Example: Probabilistic Failures

from sentinel.chaos import ToolFailureInjector
from sentinel.env import MockTool

# Create injector with 30% failure rate
injector = ToolFailureInjector(
    tool_name="api_call",
    failure_type="rate_limit",
    probability=0.3,           # 30% of calls will fail
    seed=42,                   # Deterministic
    after_step=2               # Only inject after 2 successful calls
)

# Wrap multiple tools
api_tool = MockTool("api_call", response={"status": "ok"})
wrapped = injector.wrap(api_tool)

# Test resilience
for i in range(10):
    try:
        result = wrapped(endpoint="/data")
        print(f"Call {i+1}: Success")
    except Exception as e:
        print(f"Call {i+1}: {type(e).__name__}")

Example: Custom Error Messages

injector = ToolFailureInjector(
    tool_name="database",
    failure_type="error",
    error_message="Database connection pool exhausted (max: 100)"
)

# The injected error will use your custom message
wrapped = injector.wrap(db_tool)

LLMFailureInjector

The LLMFailureInjector simulates failures at the LLM layer — rate limits, timeouts, partial responses, and stream interruptions that occur in production LLM APIs.

Failure Types

Type Description
rate_limit LLM API rate limit exceeded
timeout LLM API request timed out
partial_response Stream ended prematurely
stream_interrupt Connection lost during generation

Example: Step-Based Injection

from sentinel.chaos import LLMFailureInjector

# Inject rate limit after step 5
injector = LLMFailureInjector(
    failure_type="rate_limit",
    after_step=5,              # Only inject after step 5
    probability=0.4            # 40% chance per eligible step
)

# Check each step
for step_id in range(10):
    failure = injector.check_step(step_id)
    if failure:
        print(f"Step {step_id}: {failure['message']}")
    else:
        print(f"Step {step_id}: OK")

# Get exception for error handling
error = injector.get_error(step_id=7)
if error:
    raise error

ContextDegradation

The ContextDegradation simulator models context window erosion during long-running agent tasks. As agents execute more steps, their effective context degrades — earlier instructions get truncated, tool results accumulate noise, or the system prompt drifts from its original intent.

Strategies

Strategy Description Use Case
truncation Progressively removes earlier context entries Simulates context window pressure in long conversations
noise Injects random perturbations into tool results Models signal degradation in noisy environments
drift Gradually shifts system prompt instructions Tests agent stability when instructions change

Example: Context Truncation

from sentinel.chaos import ContextDegradation

# Configure truncation degradation
degradation = ContextDegradation(
    strategy="truncation",
    start_step=5,              # Begin degrading at step 5
    degradation_rate=0.2,      # Moderate erosion speed
    max_truncation_pct=0.5,    # Max 50% context removal
    seed=42
)

# Simulate agent context over time
context = [
    {"role": "system", "content": "You are a helpful assistant."},
    {"role": "user", "content": "What is Python?"},
    {"role": "assistant", "content": "Python is a programming language..."},
    {"role": "user", "content": "Tell me more about lists."},
]

# Process step 10 — context will be degraded
result = degradation.on_step(step_id=10, context=context)
print(f"Degradation level: {result['degradation_level']:.2f}")
print(f"Context entries: {len(result['context'])}")  # Reduced!

Example: Instruction Drift

# Drift strategy modifies the system prompt
drift_degradation = ContextDegradation(
    strategy="drift",
    degradation_rate=0.1,
    seed=42
)

system_prompt = "Always verify user identity before proceeding."

# After many steps, instructions drift
result = drift_degradation.on_step(
    step_id=20,
    system_prompt=system_prompt
)
print(f"Original: {system_prompt}")
print(f"Degraded: {result['system_prompt']}")

CascadingFailures

The CascadingFailures injector simulates error propagation in multi-agent systems. When one component fails, related components often fail shortly after — a database timeout cascades to API errors, which cascade to user-facing failures.

Example: Multi-Agent Cascade

from sentinel.chaos import CascadingFailures

# Configure cascade behavior
cascade = CascadingFailures(
    cascade_probability=0.7,   # 70% chance of cascade per failure
    max_cascade_depth=3,       # Max 3 levels deep
    propagation_delay_steps=1, # Cascade in next step
    seed=42
)

# Simulate a root failure
root_failure = {
    "tool_name": "database",
    "error_type": "timeout",
    "step_id": 1,
    "message": "Database connection timeout"
}

# Process the failure — returns cascading failures
cascades = cascade.on_failure(root_failure, current_step=1)

for c in cascades:
    print(f"Depth {c['cascade_depth']}: {c['tool_name']} "
          f"failed with {c['error_type']}")

# Check cascade summary
summary = cascade.summary()
print(f"Total cascades: {summary['total_cascades']}")
print(f"Max depth reached: {summary['max_depth_reached']}")

Cascade Patterns

Sentinel uses intelligent cascade mapping based on common failure patterns:

Root Cause Cascades To Error Type
database api_server service_unavailable
api_server user_interface service_unavailable
auth api_server service_unavailable
cache database timeout

SpecDrift

The SpecDrift injector simulates agent improvisation when operating under pressure. When agents face timeouts or resource constraints, they sometimes deviate from the original specification to "make it work" — skipping validation, using fallbacks, or substituting tools.

Drift Types

  • skip_validation — Agent skips input validation step
  • use_fallback — Uses fallback value instead of computing
  • reorder_steps — Reorders execution steps for efficiency
  • substitute_tool — Uses different tool than specified
  • truncate_output — Truncates output to save tokens
  • approximate_result — Approximates instead of precise computation

Example: Detecting Spec Drift

from sentinel.chaos import SpecDrift

# Configure drift injection
drift = SpecDrift(
    intensity="moderate",     # subtle | moderate | aggressive
    start_step=5,             # Begin checking at step 5
    probability=0.2,          # 20% base probability
    trigger_events=["timeout", "rate_limit"],  # Errors amplify drift
    seed=42
)

# Simulate agent steps with errors
recent_errors = ["timeout", "error"]

for step_id in range(1, 15):
    event = drift.check_step(step_id, recent_errors=recent_errors)
    if event:
        print(f"Step {step_id}: {event['description']}")
        print(f"  Magnitude: {event['drift_magnitude']:.2f}")
        print(f"  Cumulative: {event['cumulative_drift']:.3f}")

# Get drift assessment
score = drift.get_drift_score()
print(f"\nFinal drift score: {score['cumulative_drift']:.3f}")
print(f"Total drift events: {score['total_events']}")

NetworkPartition NEW

The NetworkPartition injector simulates network connectivity issues between agents and services. It maintains a connectivity matrix that defines which services can reach each other, allowing you to test agent behavior during partial or complete network outages.

Features

  • Connectivity Matrix — Define which services can communicate
  • Dynamic Partitioning — Add/remove partitions during test execution
  • Partial Connectivity — Simulate degraded network (high latency, packet loss)
  • Recovery Simulation — Test agent behavior when connectivity is restored

Example: Network Partition

from sentinel.chaos import NetworkPartition

# Create network partition injector
partition = NetworkPartition(
    seed=42
)

# Define initial connectivity matrix
# All services can reach each other
services = ["api_gateway", "auth_service", "database", "cache"]
partition.set_connectivity_matrix({
    "api_gateway": {"auth_service": True, "database": True, "cache": True},
    "auth_service": {"api_gateway": True, "database": True, "cache": False},
    "database": {"api_gateway": False, "auth_service": False, "cache": False},
    "cache": {"api_gateway": True, "auth_service": False, "database": False},
})

# Check if two services can communicate
can_communicate = partition.check_connectivity("api_gateway", "database")
print(f"API Gateway → Database: {can_communicate}")  # True

# Simulate partition — disconnect database from everything
partition.partition_service("database")
can_communicate = partition.check_connectivity("api_gateway", "database")
print(f"After partition: {can_communicate}")  # False

# Check which services are isolated
isolated = partition.get_isolated_services()
print(f"Isolated services: {isolated}")  # ['database']

# Restore connectivity
partition.restore_service("database")

Example: Partial Network Degradation

# Simulate high-latency network
partition = NetworkPartition(seed=42)

# Configure degradation parameters
partition.set_degradation(
    latency_ms=500,           # 500ms added latency
    packet_loss_rate=0.1,     # 10% packet loss
    jitter_ms=100             # ±100ms jitter
)

# Check degraded connection
metrics = partition.get_connection_metrics("api_gateway", "database")
print(f"Latency: {metrics['latency_ms']}ms")
print(f"Packet loss: {metrics['packet_loss_rate']*100}%")

ClockSkew NEW

The ClockSkew injector simulates time drift between distributed components. In production, servers often have slightly different system times, which can cause issues with timestamps, token expiration, session management, and distributed consensus.

Features

  • Time Drift Simulation — Add configurable offset to system time
  • Gradual Drift — Simulate clock that slowly drifts over time
  • Jump Events — Simulate sudden time changes (NTP sync, manual adjustment)
  • Component-Specific Skew — Different drift rates per service

Example: Simulating Clock Skew

from sentinel.chaos import ClockSkew
from datetime import datetime, timedelta

# Create clock skew injector
clock = ClockSkew(
    seed=42
)

# Set different time offsets for services
clock.set_skew("api_gateway", offset_ms=0)       # Reference time
clock.set_skew("auth_service", offset_ms=500)    # 500ms ahead
clock.set_skew("database", offset_ms=-200)       # 200ms behind

# Get current time for each service
gateway_time = clock.get_time("api_gateway")
auth_time = clock.get_time("auth_service")
db_time = clock.get_time("database")

print(f"API Gateway: {gateway_time}")
print(f"Auth Service: {auth_time} (+500ms)")
print(f"Database: {db_time} (-200ms)")

# Check if token will expire based on service time
token_issued = gateway_time
token_ttl = timedelta(seconds=30)
is_valid = clock.check_token_valid(
    service="auth_service",
    token_issued_at=token_issued,
    ttl=token_ttl
)
print(f"Token valid at auth_service: {is_valid}")

Example: Gradual Drift

# Configure gradual drift — clock slowly diverges
clock = ClockSkew(
    drift_rate_ms_per_second=10,  # 10ms drift per second
    seed=42
)

# Simulate time passing
for i in range(10):
    clock.advance(seconds=1)
    skew = clock.get_current_skew("database")
    print(f"Second {i+1}: {skew}ms drift")

Example: Time Jump Event

# Simulate NTP sync causing sudden time jump
clock.jump("api_gateway", offset_ms=-5000)  # Jump back 5 seconds

# Or simulate forward jump
clock.jump("database", offset_ms=30000)     # Jump forward 30 seconds

# Check for time anomalies
anomalies = clock.detect_anomalies(threshold_ms=1000)
for anomaly in anomalies:
    print(f"Anomaly: {anomaly['service']} "
          f"drifted {anomaly['skew_ms']}ms")

MemoryPressure NEW

The MemoryPressure injector simulates context exhaustion and memory constraints that agents face in production. It models scenarios where agents run out of context window space, encounter OOM errors, or need to manage limited resources.

Features

  • Context Window Exhaustion — Simulate reaching token limits
  • Memory Allocation Failures — Inject OOM-like errors
  • Garbage Collection Pressure — Model memory cleanup delays
  • Resource Contention — Simulate shared memory conflicts

Example: Context Window Exhaustion

from sentinel.chaos import MemoryPressure

# Create memory pressure injector
memory = MemoryPressure(
    max_context_tokens=8192,    # Simulated context limit
    seed=42
)

# Track token usage
for step in range(20):
    # Simulate adding context
    tokens_used = memory.add_context(
        step_id=step,
        content=f"Step {step} result data..."
    )
    
    # Check if approaching limit
    pressure = memory.get_pressure_level()
    print(f"Step {step}: {tokens_used} tokens, "
          f"pressure: {pressure:.2f}")
    
    # Check if context was evicted
    if memory.was_context_evicted(step):
        print(f"  ⚠️ Context evicted at step {step}!")

Example: Memory Allocation Failure

# Configure aggressive memory pressure
memory = MemoryPressure(
    max_context_tokens=4096,    # Low limit for testing
    eviction_strategy="lru",    # Least recently used eviction
    simulate_oom=True,          # Inject OOM errors
    oom_probability=0.1,        # 10% chance of OOM
    seed=42
)

# Try to allocate memory
try:
    memory.allocate(
        component="agent_buffer",
        size_tokens=2048
    )
except MemoryError as e:
    print(f"OOM Error: {e}")

# Check memory state
state = memory.get_memory_state()
print(f"Allocated: {state['allocated_tokens']}")
print(f"Available: {state['available_tokens']}")
print(f"Pressure: {state['pressure_level']}")

Example: Garbage Collection Pressure

# Simulate GC pauses during memory pressure
memory = MemoryPressure(
    gc_pause_ms=100,            # 100ms GC pause
    gc_frequency_steps=5,       # GC every 5 steps
    seed=42
)

# Monitor GC impact
for step in range(15):
    result = memory.process_step(step)
    if result.get("gc_triggered"):
        print(f"Step {step}: GC pause {result['gc_pause_ms']}ms")

Composing Multiple Injectors

The real power of Sentinel's chaos engineering comes from combining multiple injectors to simulate realistic production conditions. The ChaosBudget class orchestrates multiple injectors with a shared failure budget.

Example: Realistic Production Scenario

from sentinel.chaos import (
    ChaosBudget, ToolFailureInjector, ContextDegradation,
    CascadingFailures, NetworkPartition, MemoryPressure
)

# Create a realistic chaos scenario
chaos = ChaosBudget(max_failures=10)

# Layer 1: Network issues
network = NetworkPartition(seed=42)
network.set_connectivity_matrix({
    "agent": {"api": True, "database": True},
    "api": {"database": True},
})

# Layer 2: Tool failures triggered by network issues
tool_failures = ToolFailureInjector(
    tool_name="api",
    failure_type="timeout",
    probability=0.3,
    seed=42
)

# Layer 3: Context degradation under load
context_deg = ContextDegradation(
    strategy="truncation",
    degradation_rate=0.15,
    seed=42
)

# Layer 4: Cascading failures
cascade = CascadingFailures(
    cascade_probability=0.6,
    max_cascade_depth=3,
    seed=42
)

# Layer 5: Memory pressure
memory = MemoryPressure(
    max_context_tokens=8192,
    seed=42
)

# Compose everything
chaos.add(tool_failures)
chaos.add(context_deg)

# Run your agent test with chaos
print(f"Chaos budget: {chaos.remaining} failures remaining")

Chaos Presets

Sentinel includes pre-configured chaos scenarios for common failure patterns. These presets combine multiple injectors with realistic parameters.

Available Presets

Preset Description Use Case
mild_degradation Low-probability tool failures with context truncation Smoke testing, CI/CD pipelines
network_instability Intermittent network partitions and timeouts Testing retry logic and circuit breakers
memory_exhaustion Context window pressure with OOM simulation Long-running agent tasks
cascade_failure Database timeout cascading through services Testing error propagation handling
full_outage Complete service unavailability Testing graceful degradation
clock_chaos Time drift across distributed components Testing timestamp-dependent logic
production_realistic Mixed failures mirroring real production Pre-deployment validation

Example: Using Presets

from sentinel.chaos_presets import PRESETS

# Load a pre-configured chaos scenario
chaos = PRESETS["network_partition"]

# Or customize a preset
chaos = PRESETS["production_incident"]
# Wrap in a ChaosBudget for control
from sentinel.chaos import ChaosBudget
budget = ChaosBudget(max_failures=20)
for injector in chaos:
    if hasattr(injector, 'wrap'):
        budget.add(injector)

# List available presets
from sentinel.chaos_presets import PRESETS
for name, injectors in PRESETS.items():
    print(f"{name}: {len(injectors)} injectors")

Injection Records

Every chaos injection is logged in an InjectionRecord for post-hoc analysis and debugging.

Example: Analyzing Injection History

# Get all injection records
records = chaos.records

for record in records:
    print(f"[{record.timestamp:.2f}] {record.injector_type}: "
          f"{record.failure_type} → {record.target}")
    if record.message:
        print(f"  Message: {record.message}")

# Group by failure type
by_type = chaos.records_by_failure
for failure_type, records in by_type.items():
    print(f"\n{failure_type}: {len(records)} injections")

# Get summary
summary = chaos.summary()
print(f"\nChaos Summary:")
print(f"  Total injected: {summary['total_injected']}")
print(f"  Budget remaining: {summary['remaining']}")
print(f"  Injectors: {summary['injector_count']}")

Best Practices

  • Start with presets — Use pre-configured scenarios before building custom chaos
  • Use seeds for determinism — Always set seed parameter for reproducible tests
  • Set failure budgets — Prevent runaway chaos from masking test results
  • Layer injectors — Combine multiple failure types for realistic scenarios
  • Assert resilience — Use chaos assertions to validate agent recovery behavior
  • Analyze records — Review injection history to understand failure patterns
