Tool Failure Injection

Deep dive on ToolFailureInjector — simulate tool timeouts, errors, rate limits, and malformed responses.

Overview

The ToolFailureInjector is the most commonly used chaos injector in Sentinel. It intercepts calls to specific mock tools and injects realistic failures based on probability, step count, or both. This lets you verify that your agent handles degraded tools gracefully — retrying, falling back, or reporting errors — instead of silently producing wrong output.

📖 Import path from sentinel.chaos import ToolFailureInjector

Failure Types

ToolFailureInjector supports five failure types via the ToolFailureType enum. Each maps to a specific exception class:

Type HTTP Equivalent Exception Default Message
timeout 408 TimeoutError "Request timed out after 30s"
error 500 MockToolError "Internal server error"
rate_limit 429 RateLimitError "Rate limit exceeded (429)"
malformed 502 MockToolError (status 502) "Malformed response from API"
partial 200 (truncated) MockToolError (status 200) "Partial response received"

The partial type is particularly useful for testing agents that assume a 200 response means complete data. It raises a MockToolError with status 200 and a message indicating the connection was closed mid-transfer.

Constructor Parameters

ToolFailureInjector(
    tool_name: str,            # Name of the tool to target
    failure_type: str = "error",  # Type of failure (see table above)
    probability: float = 1.0,  # 0.0–1.0, chance of failure per call
    seed: Optional[int] = None,# Random seed for deterministic injection
    after_step: int = 0,       # Only inject after this many tool calls
    error_message: Optional[str] = None,  # Custom error message
)

Parameter Details

  • tool_name — The name must match the MockTool.name attribute exactly. When the injector wraps a tool with a different name, calls pass through unchanged.
  • failure_type — Accepts a string that maps to the ToolFailureType enum: "timeout", "error", "rate_limit", "malformed", or "partial".
  • probability — At 1.0 (default), every call to the targeted tool fails. At 0.1, only 10% of calls fail. Combined with seed, this makes failure injection reproducible.
  • after_step — Delays injection until the tool has been called this many times. Useful for testing agents that start clean but degrade over time.
  • error_message — Overrides the default message for the failure type. Useful for testing agents that parse error messages.

The wrap() Method

The wrap() method installs a call_handler on the MockTool so that every call to tool(**kwargs) goes through the injector's decision logic. The original tool is modified in place and also returned for convenience.

injector = ToolFailureInjector(
    tool_name="search",
    failure_type="timeout",
    probability=0.5,
    seed=42,
)

tool = MockTool("search", response="results")
injector.wrap(tool)  # tool is now wrapped

result = tool(q="query")  # may raise TimeoutError
⚠️ In-place modification wrap() modifies the tool's call_handler attribute. If you need to use the tool without injection later, create the tool before wrapping, or use a fresh instance.

Under the hood, the wrapper:

  1. Records every call as a MockToolCall (with timing)
  2. Checks if the tool name matches the target
  3. Consults the RNG to decide whether to inject (respecting after_step)
  4. If injecting: creates the configured exception, logs an InjectionRecord, and raises
  5. If not injecting: delegates to the original tool response or response function

Probabilistic Injection

When probability is less than 1.0, each tool call independently rolls against the probability. This simulates intermittent failures — the kind agents face in production where APIs are flaky but not completely down.

# 30% chance of failure on each call
injector = ToolFailureInjector(
    tool_name="database",
    failure_type="error",
    probability=0.3,
    seed=42,
)

tool = MockTool("database", response={"rows": 100})
injector.wrap(tool)

# Simulate 100 tool calls
results = []
for i in range(100):
    try:
        result = tool(query=f"SELECT * FROM users_{i}")
        results.append(("ok", result))
    except MockToolError:
        results.append(("failed", None))

failures = [r for r in results if r[0] == "failed"]
print(f"Failures: {len(failures)}/100")  # ~30 failures
print(f"Injection count: {injector.injection_count}")

Deterministic Seeding

Passing a seed ensures the same sequence of random decisions every time. This is critical for CI/CD pipelines where tests must be reproducible across runs.

# Two injectors with the same seed produce identical failure sequences
injector_a = ToolFailureInjector(
    tool_name="api",
    failure_type="timeout",
    probability=0.4,
    seed=123,
)

injector_b = ToolFailureInjector(
    tool_name="api",
    failure_type="timeout",
    probability=0.4,
    seed=123,  # Same seed = same failure pattern
)

# Both will fail on exactly the same call indices
# across any number of runs
💡 Best practice Always set a seed in test code. Use a different seed per test case so each test gets its own unique but reproducible failure pattern.

Delayed Injection with after_step

The after_step parameter lets the tool work normally for the first N calls, then begins injecting failures. This models scenarios where an agent starts with clean state but encounters failures as it performs more operations.

# Tool works fine for first 5 calls, then 50% failure rate
injector = ToolFailureInjector(
    tool_name="search",
    failure_type="rate_limit",
    probability=0.5,
    after_step=5,
    seed=42,
)

tool = MockTool("search", response="results")
injector.wrap(tool)

# Calls 1–5: always succeed
for i in range(5):
    result = tool(q=f"query_{i}")  # No injection

# Calls 6+: 50% chance of RateLimitError
for i in range(5):
    try:
        tool(q=f"query_{i+5}")
    except RateLimitError:
        print(f"Call {i+6} hit rate limit")

Combining with ChaosBudget

The ChaosBudget class caps total failures across all injectors. Use it to ensure your test doesn't get overwhelmed by injection noise.

from sentinel.chaos import ChaosBudget, ToolFailureInjector

# Create a budget: max 3 total failures across all tools
chaos = (ChaosBudget(max_failures=3)
    .add(ToolFailureInjector(
        tool_name="search",
        failure_type="timeout",
        probability=0.5,
        seed=42,
    ))
    .add(ToolFailureInjector(
        tool_name="database",
        failure_type="error",
        probability=0.8,
        seed=42,
    )))

# Each injector has its own RNG, but the budget ensures
# at most 3 failures total before the budget is exhausted
print(f"Can inject: {chaos.can_inject()}")   # True
print(f"Remaining: {chaos.remaining}")        # 3
print(f"Exhausted: {chaos.exhausted}")        # False

# After running the agent, check what happened
print(f"Total injected: {chaos.total_injected}")
print(f"Budget summary: {chaos.summary()}")

# You can also inspect records grouped by type or failure
print(f"Records by type: {chaos.records_by_type}")
print(f"Records by failure: {chaos.records_by_failure}")
⚠️ Budget exhaustion Once chaos.exhausted is True, no more failures are injected regardless of individual injector settings. Call chaos.reset() to start a new test run with a fresh budget.

The make_validator() Pattern

ToolFailureInjector supports the make_validator() pattern for use with Sentinel's assert_no_silent_failure assertion. This creates a validator function that checks whether the agent's output is still valid despite the injected failures.

from sentinel.chaos import ToolFailureInjector
from sentinel.assertions import assert_no_silent_failure

injector = ToolFailureInjector(
    tool_name="search",
    failure_type="timeout",
    probability=0.5,
    seed=42,
)

tool = MockTool("search", response="results")
injector.wrap(tool)

# Run your agent with the wrapped tool
output = run_agent(tool=tool)

# Create a validator from the injector
validator = injector.make_validator()

# Assert the agent didn't silently fail
assert_no_silent_failure(output, validator=validator)

# You can also inspect injection records for debugging
for record in injector.records:
    print(f"Injected {record.failure_type} at call {record.call_index}: {record.message}")

Properties for Post-Test Inspection

Property Type Description
injection_count int Total failures injected so far
records List[InjectionRecord] Full history of injections with timestamps, types, and messages

Complete Example

Here's a full test that validates agent resilience to tool failures:

import pytest
from sentinel.chaos import ChaosBudget, ToolFailureInjector
from sentinel.env import MockTool
from sentinel.assertions import assert_no_silent_failure, assert_graceful_degradation

def test_agent_handles_search_failures():
    """Agent should retry or fall back when search fails."""
    # Set up tools
    search_tool = MockTool("search", response="results")
    db_tool = MockTool("database", response={"rows": 10})

    # Configure chaos
    chaos = (ChaosBudget(max_failures=3)
        .add(ToolFailureInjector(
            tool_name="search",
            failure_type="timeout",
            probability=0.6,
            seed=42,
        )))

    # Wrap tools
    chaos.get_injectors()[0].wrap(search_tool)

    # Run agent
    output = run_research_agent(search=search_tool, database=db_tool)

    # Assert resilience
    assert output is not None, "Agent produced no output under failure"
    assert output != "", "Agent produced empty output under failure"

    # Verify the agent handled errors (not just lucky with non-failing calls)
    if chaos.total_injected > 0:
        assert_graceful_degradation(output)

    print(f"Handled {chaos.total_injected} failures successfully")

Full API Reference

class ToolFailureInjector:
    """Injects failures into specific mock tool calls."""

    # Constructor
    def __init__(
        self,
        tool_name: str,
        failure_type: str = "error",
        probability: float = 1.0,
        seed: Optional[int] = None,
        after_step: int = 0,
        error_message: Optional[str] = None,
    ) -> None: ...

    # Properties
    injection_count: int          # Number of failures injected
    records: List[InjectionRecord] # Injection history

    # Methods
    def wrap(self, tool: MockTool) -> MockTool:
        """Wrap a MockTool with failure injection."""

    def inject(
        self,
        tool: MockTool,
        arguments: Optional[Dict[str, Any]] = None,
    ) -> bool:
        """Manually trigger injection check."""

    def reset(self) -> None:
        """Reset injection state (clear history)."""

    # Internal
    def _should_inject(self) -> bool: ...
    def _create_failure(self, arguments: Dict[str, Any]) -> Exception: ...
    def _default_message(self) -> str: ...
