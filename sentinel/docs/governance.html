Governance

Verify agent claims against independent evidence.

Overview

Sentinel's governance harness independently verifies what an autonomous agent claims to have done. It compares the agent's diary entry (claims) against independently observed facts about the project (evidence) and produces a structured audit report with discrepancies, severity levels, and a verdict.

📖 Why governance? Autonomous agents can hallucinate — claiming tests pass when they fail, claiming files exist when they don't, or even backdating entries. Sentinel's governance harness catches these automatically, with zero trust in the agent's self-reporting.

Audit Flow

  1. Collect evidence — run tests, check file timestamps, count tools, verify existence. This happens FIRST, before reading any claims.
  2. Extract claims — parse the agent's diary entry for test counts, tool counts, files modified, dates, etc.
  3. Audit — compare claims against evidence using the deterministic comparator (always) and optionally an LLM auditor.
  4. Post-process — extract structured findings, merge results, compute final verdict.
  5. Report — save JSON + human-readable reports.

Option C: Pure Deterministic Comparator

The default and recommended approach. The deterministic comparator in extract.py is the primary audit mechanism — pure regex and quantitative comparison with zero infrastructure requirements.

✅ Why deterministic? The original build used an LLM auditor that hallucinated mismatches (claiming "actual: 0" for test counts despite evidence showing otherwise). The deterministic comparator catches all the hard errors — and it works everywhere, with zero hallucination risk.

What It Catches

Check Type Severity Description
Test count mismatch CRITICAL Agent claims N tests pass, but actual test run shows a different number.
Tool count mismatch WARNING Agent claims N tools, but @mcp.tool() decorator count differs.
Future-dated diary CRITICAL Diary claims a date in the future — catches backdating.
Backdated diary CRITICAL Diary filename claims a date, but file mtime doesn't match.
Missing files WARNING Agent claims to have modified files that don't exist on disk.
README mismatch WARNING README claims different test/tool counts than the diary.

How It Works

The comparator in extract.py runs four layers:

  1. LLM output parsing — if an LLM auditor was used, parse its JSON response and extract findings.
  2. False positive suppression — when the comparator confirms numbers match, suppress any LLM finding that claims they don't (prevents hallucinated mismatches).
  3. Deterministic checks — regex-based comparison of claimed vs actual values for test counts, tool counts, dates, and file existence.
  4. Verdict computation — any CRITICAL finding → FAIL. Any WARNING → WARN. Otherwise → PASS.

Optional LLM Auditor

For richer qualitative analysis, Sentinel supports an optional LLM auditor. The default is qwen3.5 via Ollama, though any OpenAI-compatible API works.

# auditor.yaml
backend:
  type: ollama                    # "none" (default), "openai-compatible", or "ollama"
  url: http://localhost:11434     # Ollama API endpoint

primary:
  model: qwen3.5:9b
  temperature: 0.0
  max_tokens: 1024

What the LLM Adds

  • Qualitative analysis — catches wording inconsistencies, logical gaps, and narrative drift that regex can't detect.
  • Contextual understanding — understands that "agent claimed X but evidence shows subtle Y" even when the numbers technically match.
  • Nuanced findings — produces richer descriptions of discrepancies beyond simple count mismatches.

When to Use It

  • When you want deeper analysis beyond quantitative checks
  • When your diary entries contain complex claims that need semantic understanding
  • When you're willing to accept the infrastructure cost (running a local LLM)
⚠️ Known limitation LLM auditors can produce false positives (hallucinating mismatches that don't exist). The deterministic comparator always runs as a safety net — if it confirms numbers match, any contradictory LLM finding is suppressed.

Directory Structure

The governance module lives in the governance/ directory:

governance/
├── __init__.py        # Package exports (run_audit)
├── audit.py           # Main entry point — orchestrates the full audit
├── auditor.py         # LLM auditor — calls local models via OpenAI-compatible API
├── auditor.yaml       # Auditor configuration (backend, model, retry settings)
├── claims.py          # Extract claims from diary entries (regex-based)
├── evidence.py        # Collect independent evidence (tests, files, tools)
├── extract.py         # Deterministic comparator + structured findings extraction
├── governance.yaml    # Project-level governance config
├── README.md          # Module documentation
└── reports/           # Generated audit reports (JSON + human-readable)

Key Files

  • audit.py — The main orchestrator. Runs the 5-step audit flow: collect evidence → extract claims → audit → post-process → report.
  • extract.py — The deterministic comparator. Always runs as a safety net. Catches test count mismatches, future-dated entries, missing files, and more.
  • claims.py — Parses diary markdown files and extracts verifiable claims using regex patterns (test counts, tool counts, dates, files modified).
  • evidence.py — Independently observes the project: runs tests, checks file timestamps, counts tools, analyzes README state.
  • auditor.py — Optional LLM auditor. Config-driven, supports multiple backends (Ollama, LM Studio, vLLM, any OpenAI-compatible API).

How to Run the Audit

Run the governance audit as a CLI command or as a Python library:

As a CLI

# Audit the most recent diary entry
python3 governance/audit.py . --output governance/reports/

# Audit a specific date
python3 governance/audit.py . --diary 2026-07-08 --output governance/reports/

# JSON output to stdout
python3 governance/audit.py . --json

As a Library

from governance import run_audit

# Run the full audit
result = run_audit(
    project_root=".",
    diary_date="2026-07-08",  # optional, defaults to most recent
    output_dir="governance/reports/",  # optional
)

# Inspect results
print(f"Verdict: {result.verdict.value}")
print(f"Summary: {result.summary}")
for d in result.discrepancies:
    print(f"  [{d.severity.value}] {d.type}: {d.description}")

Configuration

The governance module is configured via governance/governance.yaml and governance/auditor.yaml:

# governance/governance.yaml
diary_dir: updates          # Directory containing diary entries
test_dir: tests             # Directory to run tests from
test_command: [pytest, tests/, -v, --tb=short]
src_dir: src                # Source directory for file timestamps
source_file_pattern: "**/*.py"
mcp_server_file: null       # File to count @mcp.tool() decorators
readme_path: README.md

Severity Levels

  • CRITICAL — False claims, tests that claim to pass but failed, backdated entries, future-dated diaries. Any CRITICAL → verdict is FAIL.
  • WARNING — Count mismatches, missing files referenced in claims, README inconsistencies. Any WARNING (no CRITICAL) → verdict is WARN.
  • INFO — Minor drift, wording differences, optional claims with no evidence. Does not affect verdict.
